# Mail Lab — virtuální mail providery (1:1 jako reálné schránky)

**Status:** plánováno
**Datum založení:** 2026-04-29
**Trigger:** uživatel — "potřebujeme vytvořit 1:1 testovací prostředí, které se bude chovat naprosto identicky jako e-mailové schránky a na tomto to budeme testovat a vyvíjet, tj. vytvoříme si testovací (virtuální prostředí)."

**Supersedes (částečně):** `2026-04-29-brutal-test-harness.md` — předchozí iniciativa použila greenmail/mailpit (toy SMTP), což není 1:1. Tato iniciativa nahrazuje testovací stack za production-grade Postfix+Dovecot+Rspamd. Brutal harness scenáře zůstávají platné — jen běží proti Mail Labu místo greenmailu.

## Kontext

Dnes vyvíjíme a testujeme proti:
- **Greenmail** — Java toy SMTP+IMAP, nehraje DKIM/DMARC, nemá real spam filtering, MX lookup nefunguje
- **Mailpit** — catch-all SMTP only, IMAP minimální, žádný realistic acceptance protocol
- **Prod Seznam/Gmail** — real provider, ale jen pro post-deploy validaci. NIKDY pro dev (HARD RULE memory `feedback_no_external_services` + `feedback_campaign_send`)

Důsledky toy environment:
- "Funguje lokálně" → fail v prod (Mullvad IPs rejectovány Seznamem, my testovali proti greenmailu co accept everything)
- DKIM/SPF/DMARC selhání nikdy neviditelné v dev
- Greylisting (4xx then accept) — Seznam to dělá, my si neprocvičíme retry path
- Kvóty / rate limit per sender — Seznam má 100 mail/h limit, greenmail je nelimitovaný
- Bounce kódy a fráze — `554 5.7.1 Spam message rejected` z greenmailu nemáme, prod má

Cíl: **lokální stack co se chová neodlišitelně od reálného mail providera** pro dev + test. Operátor nastaví dashboard na `seznam.lab` místo `seznam.cz`, posílá kampaně, sleduje real-grade reakce. Test harness (#211) drovuje stejný stack.

## Měřitelné cíle

1. **3 simulovaní provideři** — `seznam.lab`, `gmail.lab`, `outlook.lab` — každý má vlastní DKIM klíč, vlastní acceptance pravidla, vlastní webmail na separátním hostnamu.
2. **Production-grade SMTP+IMAP** — Postfix (MTA) + Dovecot (IMAP IDLE/CONDSTORE/MOVE) + Rspamd (spam filter) + OpenDKIM + OpenDMARC. Plný RFC 5321/3501 compliance.
3. **Realistické provider chování** — per-provider profil (viz tabulka níže) — Seznam reject Mullvad/Tor IPs, Gmail spam-classify high link-density, Outlook greylist new senders.
4. **DNS resolver v docker net** — `unbound` s lab zónami; orchestrator vidí `MX seznam.lab → mx.seznam.lab → 10.x.x.x` correctly.
5. **Webmail per provider** — Roundcube na `webmail.seznam.lab` atd.; operátor inspectuje stav schránky stejně jako prod webmail.
6. **Persistent state** — volumes přežívají `docker compose down`. Schránky, mail, spam learning data zůstávají.
7. **Admin REST API** — Go service `features/platform/mail-lab-api` co umí: create/drop mailbox, set quota, inject inbound mail, set bounce profile, trigger greylist, set rate limit. Driver test harness mluví přes toto API.
8. **Hermeticita** — stack nemá venkovní egress (docker network `internal: true` + iptables OUTPUT DROP). Žádný unintended kontakt s prod Seznamem/Gmailem.
9. **Single command bootstrap** — `bash scripts/mail-lab/up.sh` → 5 minut a stack je healthy + seedovaný demo daty.
10. **Operator dev workflow** — `cp features/platform/outreach-dashboard/.env.lab.example features/platform/outreach-dashboard/.env.local` + `pnpm dev` → vidí dashboardy s lab providery.

Non-goals:
- Plný `mail-tester.com` skóre fidelity — chceme **chování**, ne perfektní reputační skóring (test prostředí má vždy clean rep).
- Anti-malware (ClamAV) — dnes neuměřitelný benefit pro testy.
- Multi-tenant SaaS-grade — single-host docker stack je dostatečný.

### Provider profily

Profily encoded v `features/platform/mail-lab-api/profiles/{seznam,gmail,outlook}.yaml`. Každý profil drží:

| Pole | Seznam | Gmail | Outlook |
|---|---|---|---|
| domain | seznam.lab | gmail.lab | outlook.lab |
| smtp_submission_port | 465 (SMTPS) + 587 (STARTTLS) | 465 | 587 |
| imap_port | 993 (IMAPS) | 993 | 993 |
| max_message_size | 30 MB | 25 MB | 35 MB |
| mailbox_quota | 1 GB | 15 GB | 50 MB |
| rate_limit_per_hour | 100 | 500 | 30 |
| reject_non_cz_origin | true | false | false |
| greylist_unknown_sender | false | false | true |
| spam_classify_link_ratio | 0.5 | 0.3 | 0.4 |
| reject_proxy_ips_cidr | mock list (Mullvad CIDR ranges per memory `seznam_proxy_geo_mismatch`) | — | — |
| bounce_kind_on_reject | 5.7.1 | 5.7.26 | 5.7.606 |
| dkim_required | strict | relaxed | relaxed |
| auto_reply_enabled | configurable | configurable | configurable |

Profily lze runtime přepínat přes `POST /profile/seznam.lab/override` — chaos scenário např. flipne `reject_non_cz_origin: false` aby se otestoval happy-path.

### Klíčová rozhodnutí

| Téma | Volba | Odůvodnění |
|---|---|---|
| Mail server stack | Postfix + Dovecot + Rspamd + OpenDKIM/OpenDMARC | Production-grade, RFC compliant, debian docker images stable. Neumí nás odlišit od reálu pro test účely. Alternative `mailcow` je heavyweight (10+ kontejnerů, 4GB RAM) a má SOGo místo Roundcube. |
| Bázový image | `mailserver/docker-mailserver` | Curated postfix+dovecot+rspamd image, docker-compose-ready, aktivně udržovaný. |
| Webmail | Roundcube | Lightweight, IMAP-based, SQL backend, defaultní volba pro Postfix+Dovecot setupy. |
| DNS | unbound | Lightweight resolver, `local-data:` syntax pro lab zóny. Orchestrator + relay v docker net používají tento resolver místo `8.8.8.8`. |
| Multi-provider isolation | Per-provider docker compose service group + samostatný network alias | Každý provider běží svůj postfix instance — žádné cross-contamination, lze restart/break jednoho. |
| Admin API jazyk | Go (sdílí types + lib s ostatními services) | Konzistence s monorepo. Alternative Python s `pyMailbox` lib by byl rychlejší napsat, ale pak by jeden další runtime. |
| Profile config | YAML | Operátor friendly, version controlled v repo. |
| Bounce simulation | Postfix `smtpd_relay_restrictions` + custom Rspamd rules | Real bounce kódy + frází. |
| State persistence | named docker volumes (`mail-lab-seznam-data` atd.) | Schránky přežívají `docker compose down`; `--clean` flag wipe. |
| Bootstrap data | `scripts/mail-lab/seed.sh` | Idempotent — vytvoří 1 operator account + 5 prospect účtů per provider, naseje 3 testovací thready. |

## Rizika

- **Stack heaviness** — 3 provideři × ~4 kontejnery = 12+ containers, ~3 GB RAM. Mitigation: per-provider lze spustit selektivně (`--providers=seznam`); CI run jen 1 provider unless test specifically multi-provider.
- **DKIM klíče v repo** — netradiční, ale Mail Lab je test-only; klíče jsou public, dokumentujeme to v README. **Žádné prod creds**.
- **docker-mailserver upgrade churn** — mailserver image se updatuje ~2× měsíčně. Mitigation: pin specific tag, upgrade jako separátní PR s smoke test.
- **Provider behavior fidelity** — naše profily jsou aproximace reálných providerů. Mitigation: dokumentovat known divergences v `docs/playbooks/mail-lab-quickstart.md`; sledovat real-world bounce z prod (Sentry) a iterovat profil.
- **State drift** — operator persistent volumes mají historický stav co se neshoduje s aktuálními tests. Mitigation: `up.sh --clean` wipe; `seed.sh` idempotent re-seeds.
- **Memory rule respect** — testy NIKDY non sahají na real prod creds nebo external network (memory `feedback_no_external_services`, `feedback_no_direct_smtp`, `feedback_campaign_send`). Iptables OUTPUT DROP zajišťuje hermeticita.
