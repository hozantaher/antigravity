# ADR-002: Anti-State-Level and Anti-Physical-Access Defenses

**Status:** Accepted
**Date:** 2026-04-03
**Supersedes:** ADR-001 sections 5.11, 5.12 (store-and-forward, traffic analysis resistance)
**Authors:** Spec Kit Codex Stack core council
**Scope:** `services/anti-trace-relay/internal/{constrate,pool,deaddrop,amnesic,ephemeral,duress}` + `cmd/submit/`

---

## 1. Context and Problem Statement

ADR-001 established the anti-trace relay MVP with 24 packages covering sanitization, identity separation, E2E encryption, and transport anonymization. Security audit identified two remaining adversaries that the MVP cannot adequately defend against:

### Adversary 1: State-level actor with global network visibility

The MVP's defenses (random delays, cover traffic ratio, jittered batch intervals) are statistically defeatable:

| MVP Mechanism | Weakness |
|---|---|
| Random delay 30-300s | Known uniform distribution; with weeks of samples, adversary fits arrival/departure pairs |
| Fixed cover ratio 30% | Adversary subtracts 30% and analyzes remainder |
| Jittered batch intervals | Periodicity creates spectral signature detectable via Fourier analysis |
| Single relay path | One network tap captures full traffic matrix |
| Direct delivery | Sender-recipient link observable at exit |

### Adversary 2: Physical access to submitter's device

The MVP has no client-side component. If an adversary seizes the submitter's device:

| Gap | Impact |
|---|---|
| No amnesic mode | Submission history on disk |
| No secure memory | Keys persist in RAM/swap after use |
| No duress mechanism | No defense against coercion |
| Long-running process | Extended window for memory forensics |

### Design Principle

**"Nothing to find" is stronger than "something hidden."**

- Constant-rate traffic **eliminates** patterns rather than hiding them
- Amnesic clients **eliminate** state rather than encrypting it
- Dead drops **eliminate** sender-recipient links rather than obscuring them

This principle guides every decision in this ADR.

## 2. Decision Drivers

| Driver | Weight | Rationale |
|--------|--------|-----------|
| Eliminate traffic volume analysis | Critical | Volume is the most powerful signal for state-level adversaries |
| Decorrelate message ordering | Critical | Sequential correlation defeats random delays |
| Remove sender-recipient link | Critical | Direct delivery is the simplest attack vector |
| Zero persistent state on client | Critical | Nothing on disk = nothing to find |
| Coercion resistance | High | Adversaries in conflict zones use physical coercion |
| Secure memory management | High | Keys in RAM/swap are extractable |
| Go stdlib only | High | Consistency with ADR-001 constraint |
| Integration with existing pipeline | Medium | Must work with existing 24 packages |

## 3. Considered Alternatives

### Against state-level traffic analysis

| Alternative | Verdict | Rationale |
|---|---|---|
| **Constant-rate emission (Loopix model)** | **Accepted** | Eliminates volume analysis entirely |
| **Pool-based mixing** | **Accepted** | Decorrelates order; anonymity set = pool size |
| **Dead drop bulletin board** | **Accepted** | Eliminates sender-recipient link |
| Multi-path splitting (Shamir) | Rejected | Requires multiple independent servers; violates single-binary constraint |
| Steganographic channels | Rejected | Requires TLS internals Go stdlib doesn't expose; fragile |
| Nym/HOPR mixnet | Rejected | External dependency, token economics, operational complexity |

### Against physical device access

| Alternative | Verdict | Rationale |
|---|---|---|
| **Amnesic submission** | **Accepted** | Zero persistent state; everything from passphrase |
| **Ephemeral mode (mlock + wipe)** | **Accepted** | Prevents key leakage to swap/disk |
| **Duress system** | **Accepted** | Forensically indistinguishable from wrong password |
| Plausible deniability storage | Rejected | Requires custom filesystem; thousands of lines, huge attack surface |
| Steganographic file storage | Rejected | Image codec manipulation is research-grade, not engineering |
| Full-disk encryption | Out of scope | OS-level concern, not application-level |

## 4. Architecture: Anti-State-Level Defenses

### 4.1 Constant-Rate Emission (`internal/constrate/`)

**Problem:** Variable-rate output leaks traffic volume.

**Solution:** `Emitter` sends exactly one message per fixed interval. When real messages exist in the pool, it sends one. When empty, it sends cover traffic. The observer sees an identical, steady stream regardless of actual submission volume.

```
Time:    |--5s--|--5s--|--5s--|--5s--|--5s--|--5s--|
Output:  [real] [covr] [covr] [real] [covr] [real]
         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
         Observer sees: constant stream, 12 msg/min
         Cannot distinguish: 3 real messages from 6 total
```

**Key design decision:** Jitter is intentionally **zero**. The constant cadence IS the defense. Any variation is a signal.

**Interfaces:**

```go
type MessageSource interface {
    Draw() (model.Envelope, bool)  // bool = isReal
}

type Sender interface {
    Send(ctx context.Context, env model.Envelope) error
}
```

**Integration:** Replaces the existing batch drain loop + jitter in `main.go`. The `MixPool` implements `MessageSource`.

**File:** `internal/constrate/emitter.go`

### 4.2 Pool-Based Mixing (`internal/pool/`)

**Problem:** Even at constant rate, if messages exit in roughly the same order they entered, the adversary correlates by sequence position.

**Solution:** Messages enter a `MixPool` of minimum size N. On each emitter tick, one message is selected **uniformly at random** using `crypto/rand`. The probability of correctly matching an input to an output is 1/N.

```
Input order:   A  B  C  D  E  F  G  H  (pool min=5)
Pool state:    [A,B,C,D,E] -> draw random -> output D
               [A,B,C,E,F] -> draw random -> output B
               [A,C,E,F,G] -> draw random -> output F
                            ...
Output order:  D  B  F  A  G  C  E  H  (fully decorrelated)
```

**Anonymity set guarantee:** If `pool.Size() < minSize`, `Draw()` returns cover traffic instead of a real message. This ensures the anonymity set is never smaller than `minSize`.

**Selection algorithm:** Uniform random via `crypto/rand`, swap-remove for O(1) extraction. NOT `math/rand` (predictable seed).

**File:** `internal/pool/pool.go`

### 4.3 Dead Drop Bulletin Board (`internal/deaddrop/`)

**Problem:** Even with perfect mixing and constant rate, direct delivery reveals the recipient endpoint.

**Solution:** Instead of delivering to a recipient, the relay posts messages to opaque **dead drop slots**. The recipient polls their slot independently. No observable causal link between sender and recipient.

**Slot ID derivation:**

```
SlotID = HMAC-SHA256(sharedSecret, "deaddrop-slot" || epochBytes)
```

- `sharedSecret`: 32 bytes, shared out-of-band between sender and recipient
- `epoch`: hours since Unix epoch (rotates hourly to prevent long-term correlation)
- The relay never sees the shared secret; it sees only opaque 32-byte slot IDs

**Properties:**
- Without the shared secret, slot IDs are random noise
- Hourly rotation prevents long-term access pattern correlation
- Recipients poll with cover polls to random slots to hide access patterns
- Slots expire after configurable TTL (default 24h)

**API:**

```
POST /v1/drop/{slotID}  -- post ciphertext (unauthenticated)
GET  /v1/drop/{slotID}  -- poll and remove messages (unauthenticated)
```

**Why unauthenticated:** Authentication would create a link between the slot and an identity. The slot ID itself is the credential -- knowing it (derived from the shared secret) is proof of authorization.

**File:** `internal/deaddrop/store.go`

### 4.4 Combined Flow (Before vs. After)

**Before (ADR-001):**

```
Submit -> Sanitize -> Seal -> Random delay 30-300s -> Batch + 30% cover -> Shuffle -> Direct delivery
                                                      ^^^^^^^^^^^^^^^^     ^^^^^^^     ^^^^^^^^^^^^^^^
                                                      Volume leaks         Weak mix    Recipient visible
```

**After (ADR-002):**

```
Submit -> Sanitize -> Seal -> MixPool (min N=20) -> Constant-rate emit -> Dead drop slot
                              ^^^^^^^^^^^^^^^^^     ^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^
                              1/N correlation       Zero volume signal    No recipient link
```

## 5. Architecture: Anti-Physical-Access Defenses

### 5.1 Amnesic Submission (`internal/amnesic/`)

**Problem:** Any persistent state on the device is evidence.

**Solution:** Everything the client needs is derived from a memorizable passphrase. The passphrase exists only in the user's memory. The binary itself is a generic Go executable.

**Derivation chain:**

```
passphrase
    |
    v
PBKDF2-HMAC-SHA256(passphrase, "anti-trace-amnesic-v1", 600_000 iterations)
    |
    v
master key (64 bytes)
    |
    +--> HKDF-Expand(master, "encryption") --> 32-byte AES key
    |
    +--> HKDF-Expand(master, "signing")    --> Ed25519 private key
    |
    +--> HMAC-SHA256(master, "deaddrop-slot" || epoch) --> SlotID
```

**600,000 PBKDF2 iterations:** At ~28s on modern hardware (measured in tests), this makes brute-force of even weak passphrases take years. Matches OWASP recommendation for high-security applications.

**Deterministic property:** Same passphrase always produces same keys and slot ID (for the same epoch). This means the sender and recipient can independently derive the same slot ID from the same shared passphrase without any prior state exchange.

**Zero-state lifecycle:**

```
1. Read passphrase from stdin (echo disabled)
2. Derive all keys in SecureBuffers (mlocked)
3. Encrypt message with recipient's X25519 key
4. Pad to size class
5. Post to dead drop slot
6. Zero all SecureBuffers
7. Zero passphrase
8. Exit process
Total runtime: < 30 seconds
```

**Files:** `internal/amnesic/derive.go`, `internal/amnesic/submit.go`

### 5.2 Ephemeral Memory Management (`internal/ephemeral/`)

**Problem:** Go's garbage collector does not zero freed memory. Keys persist in heap/swap indefinitely.

**Solution:** `SecureBuffer` -- a byte slice that is:

1. **mlocked** via `syscall.Mlock()` -- pinned in RAM, never swapped to disk
2. **Zeroed on free** via explicit `Zero()` method
3. **Safety-netted** via `runtime.SetFinalizer` -- zeroed if GC collects without explicit free
4. **Registered** in global registry -- `WipeAll()` zeros everything in emergency

**Signal-safe cleanup (`process.go`):**

```
SIGTERM / SIGINT / SIGHUP  -->  cleanup()  -->  WipeAll()  -->  os.Exit(0)
Panic                      -->  cleanup()  -->  WipeAll()  -->  re-panic
```

**Known limitation:** `SIGKILL` cannot be caught. Defense: the amnesic client runs for seconds, not hours. The window between key derivation and zeroing is minimal.

**Files:** `internal/ephemeral/memory.go`, `internal/ephemeral/process.go`

### 5.3 Duress System (`internal/duress/`)

**Problem:** Under coercion, the submitter may be forced to demonstrate the system.

**Solution:** The user memorizes **two** passphrases:

```
Real:   "correct horse battery staple"
Duress: "correct horse battery saddle"
```

Both derive valid-looking identities via `amnesic.Derive()`. The duress identity's Ed25519 public key is **not registered** with the relay. When the client attempts to authenticate, the relay rejects the duress key.

**Forensic indistinguishability:**

| | Real passphrase | Duress passphrase | Wrong passphrase |
|---|---|---|---|
| Derives keys? | Yes | Yes | Yes |
| Keys look valid? | Yes | Yes | Yes |
| Auth succeeds? | Yes | No | No |
| Client behavior on failure | N/A | "Authentication failed" | "Authentication failed" |
| State after failure | N/A | Zeroed | Zeroed |

The adversary watching over the user's shoulder cannot distinguish duress from a typo. There is no separate code path, no duress flag, no hidden menu. The binary behaves identically in both cases.

**Design constraint:** The client binary itself must contain zero evidence of a duress mechanism. There is no `isDuress` variable, no `duressHandler` function. The duress response is simply the normal response to auth failure: wipe and show a generic error.

**File:** `internal/duress/detector.go`

### 5.4 Amnesic Client Binary (`cmd/submit/`)

**Problem:** The submitter needs a usable tool that embodies all the above protections.

**Solution:** A single Go binary that:

- Reads passphrase from stdin with echo disabled
- Accepts relay URL and recipient key as args or env vars
- Runs the amnesic Submit workflow
- Exits in under 30 seconds
- Leaves zero trace on disk

```bash
echo "my passphrase" | ./submit \
  --relay https://relay.example.com \
  --recipient-key <hex-x25519-pubkey> \
  --message "I need help getting out"
```

**Build:** `go build -ldflags "-s -w" -o submit ./cmd/submit/`
- `-s -w`: strips debug info and symbol table
- Binary name is generic (rename to `update-helper` or `sync-agent` for operational use)

**File:** `cmd/submit/main.go`

## 6. Detailed Technical Decisions

### D14: Constant-rate over jittered batching

**Decision:** Fixed-interval emission with zero jitter.

**Rationale:** Jitter helps against passive timing analysis but fails against statistical analysis. With enough samples, jitter is just noise around a detectable signal. Constant rate eliminates the signal entirely -- there is nothing to detect. The emission interval is a system constant, not a random variable.

**Trade-off:** Constant rate means the system emits even when idle. This costs bandwidth but provides information-theoretic privacy rather than computational privacy.

### D15: Pool mixing over random delay

**Decision:** Messages enter a pool of minimum size N and exit in uniformly random order.

**Rationale:** Random delay preserves message ordering (first-in-first-out with noise). Pool mixing destroys ordering entirely. The anonymity set is bounded by pool size, not delay distribution.

**Configuration:** `MIX_POOL_MIN_SIZE` (default: 20). Larger pools provide stronger anonymity but increase latency. For a system with 100 submissions/day, a pool of 20 means each message waits an average of ~5 hours (at 12 emissions/min from constant rate).

### D16: Dead drops over direct delivery

**Decision:** Messages posted to opaque HMAC-derived slots, polled by recipients.

**Rationale:** Direct delivery creates a causal link observable by any network tap. Dead drops break this link entirely -- the relay does not know which slot belongs to which recipient. The slot ID is a shared secret between sender and recipient; the relay sees only random-looking 32-byte identifiers.

**Epoch rotation:** Slots rotate hourly. This prevents long-term correlation of polling patterns to slot IDs. The shared secret is permanent; the slot ID is ephemeral.

### D17: PBKDF2 with 600K iterations over Argon2

**Decision:** PBKDF2-HMAC-SHA256 with 600,000 iterations.

**Rationale:** Argon2id is stronger (memory-hard) but requires `golang.org/x/crypto`, violating the stdlib-only constraint. PBKDF2 is available via `crypto/hmac` + `crypto/sha256`. 600K iterations provide ~28s derivation time, making brute-force of even 4-word passphrases take centuries.

**Post-MVP:** When the stdlib constraint is relaxed, switch to Argon2id with 64MB memory, 4 iterations, 4 parallelism.

### D18: mlock + WipeAll over standard GC

**Decision:** Explicit memory management for cryptographic material.

**Rationale:** Go's GC does not zero freed memory. A heap scan (memory dump, cold boot attack) can extract keys minutes or hours after they were "freed." `mlock` prevents swap, explicit `Zero()` overwrites, and `WipeAll()` provides a single emergency kill switch.

**Known limitation:** Go's escape analysis may copy SecureBuffer contents to the stack. This is mitigated by keeping sensitive operations within the SecureBuffer's methods rather than extracting bytes into local variables.

### D19: Auth-failure-as-duress over explicit duress flag

**Decision:** Duress is indistinguishable from wrong password by design.

**Rationale:** An explicit duress mechanism (hidden flag, secret menu, separate binary) is detectable by forensic analysis. Our approach has no forensic footprint because the duress path IS the error path. There is no code difference between "wrong password" and "duress password" -- both result in auth failure, state wipe, and a generic error message.

**Operational implication:** The user must accept that using the duress passphrase destroys their ability to submit through that identity. The duress passphrase is a one-shot defense, not a decoy mode.

### D20: Unauthenticated dead drop endpoints

**Decision:** `POST /v1/drop/{slotID}` and `GET /v1/drop/{slotID}` require no authentication.

**Rationale:** Authentication creates a linkable identity. The slot ID itself is the credential -- derived from a shared secret via HMAC-SHA256, it is computationally infeasible to guess. Rate limiting is applied per-IP at the transport level (Tor provides this implicitly). Slot size limits prevent abuse.

**Trade-off:** Without auth, anyone who discovers a slot ID can post garbage to it. Mitigation: slots have max message count (default 100) and TTL (default 24h). The recipient's polling reveals nothing because they also poll random decoy slots.

## 7. Updated Threat Model

### State-level adversary (before ADR-002 vs. after)

| Attack Vector | Before | After | Mechanism |
|---|---|---|---|
| Volume analysis | Defeatable in weeks | **Eliminated** | Constant-rate emission |
| Order correlation | Defeatable with effort | **1/N probability** | Pool mixing (N=20) |
| Timing correlation | Partially resistant | **Eliminated** | Fixed interval, no jitter |
| Sender-recipient link | Observable | **Eliminated** | Dead drop slots |
| Batch interval detection | Fourier-detectable | **Eliminated** | No batching, constant rate |
| Cover traffic subtraction | 30% constant, subtractive | **Not applicable** | Cover indistinguishable from real at constant rate |

### Physical access (before ADR-002 vs. after)

| Attack Vector | Before | After | Mechanism |
|---|---|---|---|
| Disk forensics | Full history visible | **Nothing to find** | Amnesic client, zero state |
| Memory forensics (running) | Keys in heap | **mlocked + zeroed** | SecureBuffer, WipeAll |
| Memory forensics (swap) | Possible | **Prevented** | mlock prevents swap |
| Coercion | No defense | **Duress path** | Indistinguishable from wrong password |
| Runtime window | Hours | **Seconds** | One-shot submit, immediate exit |
| Binary identification | N/A | **Partially mitigated** | Stripped symbols, generic name |

### Remaining weaknesses (honest assessment)

| Weakness | Why it remains | Mitigation path |
|---|---|---|
| Intersection attack | Small user base = small anonymity set | Grow user base; pool size helps but doesn't solve |
| SIGKILL memory capture | OS cannot catch SIGKILL | Minimize runtime window (seconds) |
| Cold boot on running relay | RAM freezing extracts keys | Hardware security (HSM) post-MVP |
| Binary analysis | Go binaries have identifiable structure | `garble` obfuscator (outside stdlib) |
| OS-level logs | DNS, conntrack, journal | Operational: use Tails, disable logging |
| Relay compromise | Adversary modifies binary | Multi-relay mix network (future) |
| Rubber-hose | Physical coercion for passphrase | Duress mitigates; cannot fully solve |
| Long-term statistical analysis with large corpus | Months of constant-rate data reveals capacity | Rotate emission rate; multiple relays |

## 11. Verification

### New automated tests (25 tests across 4 packages)

| Package | Tests | Coverage Focus |
|---------|-------|----------------|
| `pool` | 5 | Submit/Draw, cover below minSize, random draw, order decorrelation, empty pool |
| `deaddrop` | 9 | Post/Poll, empty slot, TTL expiry, slot full, payload size limit, GC, deterministic SlotID, epoch rotation, different secrets |
| `amnesic` | 5 | Deterministic derivation, different passphrases, key lengths, zero wipes keys, passphrase zeroed after derive |
| `ephemeral` | 6 | Alloc, write/read, zero, double-zero safety, WipeSlice, WipeAll registry |

### Key assertions

1. **Constant-rate:** Emitter outputs exactly 1 message per interval regardless of input volume
2. **Pool mixing:** Output order != input order (statistical test with n=50, p < 10^-30 for accidental match)
3. **Dead drop:** SlotID is deterministic from (secret, epoch) and different across secrets/epochs
4. **Amnesic:** Same passphrase = same keys; different passphrase = different keys
5. **Ephemeral:** SecureBuffer bytes are all zero after `Zero()`
6. **Passphrase wipe:** Input passphrase is zeroed by `Derive()` (caller's slice modified)

## 12. Risks and Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Emission rate reveals system capacity | Medium | Adversary knows max throughput | Use multiple relays at different rates |
| Pool starvation (low traffic = long latency) | Medium | Messages wait hours in pool | Accept latency trade-off; minSize is configurable |
| Dead drop abuse (slot filling) | Low | Denial of service per slot | Max slot size (100), TTL (24h), per-slot limits |
| PBKDF2 too slow for users | Low | UX friction | 28s is acceptable for security context; Argon2 post-MVP |
| Passphrase weakness | Medium | Brute-force if adversary has epoch + slot | 600K iterations makes even 4-word phrases take years |
| mlock fails (unprivileged) | Medium | Keys may reach swap | Log warning; operational guide recommends privileged deployment |
| Cover traffic bandwidth cost | Low | Constant emission costs bandwidth | At 5s intervals with 32KB max, worst case ~6KB/s = trivial |

## 14. Relationship to ADR-001

ADR-002 **extends** ADR-001. It does not replace it.

| ADR-001 Component | ADR-002 Impact |
|---|---|
| Intake pipeline (sanitize, seal) | Unchanged |
| Identity vault | Unchanged |
| Metadata minimization | Unchanged (bucketing + padding still apply) |
| Content encryption (X25519+HKDF+AES-256-GCM) | Unchanged; reused by amnesic client |
| Message bus | Unchanged; feeds MixPool instead of relay scheduler |
| **Store-and-forward relay** | **Superseded** by constant-rate emitter + pool |
| **Traffic shaping (cover + shuffle)** | **Superseded** by constant-rate emission |
| Transport chain (Tor/VPN) | Unchanged |
| Audit trail | Unchanged |
| Exit boundary | Extended with dead drop as additional exit type |
| **No client-side component** | **Extended** with amnesic submit binary |

### Migration path

The existing `traffic/` and `relay/` packages remain functional for the `record-only` and `bridge` delivery modes. The new `constrate/` + `pool/` are used when `DELIVERY_MODE=deaddrop`. Both paths coexist.

## 15. What This System Is and Is Not

### What it is

A privacy-hardened relay that makes surveillance **expensive and unreliable**. Against regional adversaries (local ISP, national telecom, police without state-level resources), the system provides strong protection: no volume signal, no ordering signal, no sender-recipient link, no device evidence, and a coercion defense.

### What it is not

An absolute anonymity guarantee. Against a resourced state-level adversary with:
- Global network visibility across all ISPs
- Months of patience for statistical analysis
- Physical access to both the relay server AND submitter device
- Willingness to use physical coercion

...no technical system can provide absolute protection. The system raises the cost and reduces the reliability of surveillance, but cannot make it impossible.

### Honest positioning

**Say:** "Privacy-hardened relay with constant-rate emission, pool mixing, dead drops, and amnesic clients."

**Don't say:** "Untraceable." "Anonymous." "NSA-proof." "Impossible to identify."
