## Rollback decision tree

Trigger podmínky pro pause + rollback:

- **CRITICAL:** delivery rate <50% during first 4 hours (T0 to T+4h). Pause campaign 1, investigate sanitizer-fix-deploy verification.
- **HIGH:** error rate >10/min v Sentry pro 15+ minutes. Pause + check Sentry stack traces.
- **HIGH:** mailbox circuit_opened_at populated (auth fail or quota exhaustion). Pause + manual unblock.
- **MEDIUM:** Mullvad pool ok_count = 0 (both CZ + DE down). Pause + Railway intervention.
- **LOW:** Spam classification rate >20% (recipient-side feedback unavailable directly, monitor proxy via watchdog scoring drops).

Rollback steps:

1. `UPDATE campaigns SET status='paused' WHERE id=1;` — okamžité.
2. Check relay `/v1/status` queue_depth — wait for drain (in-flight envelopes deliver).
3. Investigate root cause via Sentry + slog records.
4. Document v post-mortem.
5. Decide: revert PR (e.g., #723 if sanitizer fix caused regression) vs config change (e.g., revert SMTP_HELLO_DOMAIN env).

## Open questions

První je rate limit u Seznam SMTP. 200 envelope kampaň 1 znamená do 200 sendů z 4 mailboxů (50 per mailbox at peak). Daily cap 120 per mailbox stačí. Ale Seznam může mít hidden per-second rate limit pro single sender authentication. Doporučení: P4.1 sleduje first-hour pacing — pokud Seznam vrací 421 throttle, pace down via `MIN_DELAY_SECONDS` env.

Druhá je template name resolution. Sequence config má `intro_machinery` ale disk má `heavy-01-intro.tmpl`. Pokud resolver mapuje (alias z `template.go` comment), funguje. Pokud ne, kampaň 1 selže na render. P1.7 verifikuje. Pokud broken, fallback: rename disk file nebo update sequence_config v DB.

Třetí je `pnpm verify:launch` BFF dependency. Gates 1+2 vyžadují BFF na :18001. Pokud operator BFF nestartuje lokálně před P3.2, verify-launch dry-run selže na první dvou gates. Workaround: spustit BFF lokálně 5 min před verify nebo spoléhat na production BFF deployment status.

Čtvrtá je relay observability gap. Per monitoring agent finding, relay nemá Sentry ani Prometheus `/metrics`. Pokud cokoliv selže během P4 first-hour, debugging je přes ad-hoc curl + log inspection. Post-launch P5 priority pro retrofit.

Pátá je Day 1 pacing. Doporučení launch 5/d/mb = 20 envelope total Day 1. Při 200 enrollovaných contacts to znamená Day 1 reaches contacts 1-20, Day 2 contacts 21-50, Day 3 contacts 51-110, Day 4 contacts 111-200. Final hit Day 4-5 evening.

Šestá je rollback simplicity. `UPDATE status='paused'` musí být operator-runnable bez engineering present. SQL command + DATABASE_URL z .env musí být v operator notes.
