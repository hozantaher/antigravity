# ADR-003: Advanced Anonymization — Forward Secrecy, Shamir Splitting, Decoys

**Status:** Accepted
**Date:** 2026-04-03
**Supersedes:** ADR-002 sections on static recipient keys
**Authors:** Spec Kit Codex Stack core council
**Scope:** `internal/{epochkeys,shamir,fragment,decoy,pool/persist,constrate,relay/multipath}`

---

## 1. Context

ADR-002 introduced constant-rate emission, pool mixing, dead drops, and amnesic clients. Security audit identified 6 remaining gaps exploitable by a state-level adversary with 6 months of observation:

| Gap | Risk |
|-----|------|
| Static recipient X25519 key | Compromised passphrase → retroactive decryption of entire history |
| No decoy dead drop traffic | Slot occupancy patterns observable |
| Non-persistent pool | Restart reveals message residency timing |
| Silent emission failure | Constant-rate breaks on network errors |
| No message fragmentation | Size correlation possible |
| Single relay point | One network tap captures all traffic |

## 2. Decisions

### D23: Epoch-based key ratcheting (`epochkeys/ratchet.go`)

**Problem:** Recipient key derived from passphrase is static forever.

**Solution:** Derive X25519 key pair per epoch (hour):
```
HKDF-Expand(PBKDF2(passphrase), "x25519-" || epoch_bytes, 32)
```

Recipient tries current epoch AND previous epoch (for in-flight messages). Old keys are wiped — messages from past epochs become undecryptable.

**Impact:** Compromised passphrase now only exposes current + previous hour, not entire history.

**Backward compatibility:** Receive falls back to legacy `DeriveX25519KeyPair` for pre-epoch messages.

### D24: Shamir Secret Sharing (`shamir/shamir.go`)

**Problem:** Each message is a single unit — capture it, capture the content.

**Solution:** GF(256) polynomial secret sharing. `Split(secret, K, N)` produces N shares where any K reconstruct. Single share reveals zero information (information-theoretic security, not just computational).

**Parameters:** Default K=2, N=3 (3x bandwidth cost, any 2 shares reconstruct).

### D25: Message fragmentation (`fragment/fragment.go`)

**Problem:** Messages posted to single dead drop slot.

**Solution:** Split sealed message into K-of-N Shamir shares. Each share posted to a different slot:
```
SlotID[i] = HMAC-SHA256(sharedSecret, "deaddrop-fragment" || epoch || index)
```

Recipient derives all N slot IDs, polls each, reconstructs from any K.

### D26: Multi-path routing (`relay/multipath.go`)

**Problem:** All shares go to same relay — single point of compromise.

**Solution:** Round-robin distribution: share[i] → relay[i % len(relays)]. No single relay sees enough shares. Config: `RELAY_URLS=url1,url2,url3`.

### D27: Decoy dead drop posts (`decoy/poster.go`)

**Problem:** Slot occupancy patterns reveal which slots are active.

**Solution:** Each real post accompanied by N decoy posts to random slots. Decoys sealed with throwaway X25519 keys — structurally identical, undecryptable.

### D28: Persistent encrypted pool (`pool/persist.go`)

**Problem:** Relay restart empties pool, revealing message residency window.

**Solution:** Pool serialized to AES-256-GCM encrypted file after each Submit/Draw. Restored on restart.

### D29: Emission retry with constant-rate guarantee (`constrate/emitter.go`)

**Problem:** `Send()` failure silently skips emission tick — rate drops.

**Solution:** 3 retries per tick. If all fail, requeue real message, emit cover instead. Invariant: every tick = exactly 1 outbound message.

## 5. Threat Model Update

| Attack | v0.1.0 | v0.2.0 (ADR-002) | v0.2.0+ (ADR-003) |
|--------|--------|-------------------|---------------------|
| Retroactive decryption | Entire history | Entire history | **Current + previous hour only** |
| Slot occupancy analysis | Observable | Constant-rate helps | **3x decoy noise** |
| Restart timing | Pool lost | Pool lost | **Pool persists** |
| Emission rate drop | Visible | Jittered but visible | **Always constant (retry+cover)** |
| Message capture | Full content | Full content | **K-1 shares = zero information** |
| Single relay compromise | All traffic | All traffic | **Max 1 share of K needed** |

## 7. What Cannot Be Solved

| Limitation | Why |
|------------|-----|
| Brute-forced passphrase → all epoch keys | PBKDF2 600K slows but doesn't prevent |
| Multi-relay requires independent operators | Same operator = no fragmentation benefit |
| Shamir 3x bandwidth | Information-theoretic cost, cannot compress |
| Current epoch still decryptable | Forward secrecy protects past, not present |

## 8. Verification

- [x] Epoch key determinism: same passphrase + epoch = same keys (5 tests)
- [x] Epoch key rotation: different epoch = different keys
- [x] Shamir split/combine: K-of-N reconstruction (7 tests)
- [x] Shamir single share: reveals nothing about secret
- [x] E2E epoch keys: submit → relay → receive without explicit recipient key
- [x] Emission retry: constant rate maintained on mock sender failure
- [x] All 24 test packages pass with -race detector
